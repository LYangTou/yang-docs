# Java 包管理与 Maven 学习笔记

> Java 包管理解决的是“程序运行时如何找到需要的类库，以及如何处理第三方依赖冲突”的问题。Maven 在此基础上进一步提供了依赖管理、项目结构约定、构建生命周期、插件机制和仓库体系，是 Java 项目中最常见的构建与依赖管理工具之一。

## 先看结论

- JVM 执行的是类的字节码，运行过程中遇到新类时，需要知道从哪里加载这个类。
- `Classpath` 用来告诉 JVM 到哪些位置寻找类。
- 包管理的本质是管理第三方类库的位置、版本和依赖关系。
- 没有自动化包管理时，开发者需要手动下载 jar 包、手动配置编译和运行路径。
- Maven 通过“约定优于配置”降低项目配置成本。
- Maven 使用 `groupId`、`artifactId`、`version` 三个坐标唯一标识一个依赖。
- Maven 支持传递性依赖管理，但也可能出现依赖冲突。
- 常见依赖问题包括 `ClassNotFoundException`、`NoClassDefFoundError`、`AbstractMethodError`、`LinkageError`。
- Maven 不只是包管理工具，也是自动化构建工具。

## 内容结构

这部分知识可以按下面的顺序理解：

1. JVM 如何加载类，以及为什么需要 `Classpath`。
2. Java 包和第三方 jar 包如何参与程序运行。
3. 手动管理 jar 包为什么容易失控。
4. Maven 如何通过仓库、坐标和约定解决依赖管理问题。
5. Maven 如何处理传递性依赖、作用域和依赖冲突。
6. Maven 作为构建工具，还负责测试、打包、插件执行和多模块管理。

## JVM、类和 Classpath

JVM 的基本工作可以简化为：

```text
执行一个类的字节码
如果执行过程中遇到新的类，就加载这个类
继续执行
```

关键问题是：JVM 到哪里加载这些类？

这就需要 `Classpath`。

`Classpath` 表示类路径，用来告诉 JVM 可以从哪些位置寻找类。它可以指向：

- 编译后的 `.class` 文件目录
- jar 包
- 多个目录或 jar 包组成的路径集合

常见命令形式：

```bash
java -classpath target/classes;lib/mysql-connector-j.jar com.example.Main
```

也可以写成：

```bash
java -cp target/classes;lib/mysql-connector-j.jar com.example.Main
```

在 Windows 中，classpath 多个路径之间通常用分号 `;` 分隔；在 Linux/macOS 中通常用冒号 `:` 分隔。

## 包和全限定类名

Java 中的类通常会声明包名：

```java
package com.example.service;

public class UserService {
}
```

这个类的全限定类名是：

```text
com.example.service.UserService
```

全限定类名由包名和类名组成，它是 JVM 识别类的重要依据。对应到目录结构通常是：

```text
com/example/service/UserService.class
```

因此，包不仅是代码组织方式，也会影响类文件在目录或 jar 包中的位置。

## jar 包是什么

jar 包可以理解为把许多 `.class` 文件和资源文件打包在一起的压缩包。

常见用途：

- 发布 Java 类库。
- 复用第三方代码。
- 把应用程序及其资源打包交付。

例如，一个 MySQL 驱动 jar 包中包含了 JDBC 驱动相关的类。Java 程序运行时，只要这个 jar 包在 `Classpath` 中，JVM 就可以加载其中的类。

## 包管理解决什么问题

包管理的核心问题是：

```text
如何让 JVM 找到程序运行所需的第三方类库
如何管理这些类库的版本
如何处理类库之间的依赖关系
如何解决多个版本或同名类造成的冲突
```

当项目很小时，手动下载几个 jar 包还能接受。

但真实项目中，一个依赖可能还依赖其他依赖，这种关系称为传递性依赖。

例如：

```text
项目 A 依赖 B
B 又依赖 C 和 D
D 又依赖 E
```

如果全部手动下载和配置，很快就会变得难以维护。

## Classpath Hell

`Classpath hell` 指由于类路径配置复杂、jar 包版本冲突或同名类冲突导致的一类问题。

典型场景：

- 同一个类在多个 jar 包中都存在。
- 同一个 jar 包的多个版本同时出现在 classpath 中。
- 编译时使用了一个版本，运行时加载到了另一个版本。
- 某个依赖缺失，导致运行时找不到类。

由于全限定类名是类的重要标识，当多个同名类同时出现在 `Classpath` 中，JVM 最终加载哪个类会受类路径顺序等因素影响，问题通常不直观。

## 没有 Maven 时的依赖管理方式

在没有 Maven 等工具时，常见做法包括：

- 手动下载 jar 包。
- 把 jar 包放入项目的 `lib` 目录。
- 手写编译命令或运行命令。
- 手动维护 classpath。
- 使用 Ant 这类工具编写 XML 构建脚本。

Ant 能自动执行编译、复制、打包等任务，但依赖 jar 包通常仍需要手动准备。

这种方式的问题是：

- 每个项目都需要维护一套构建脚本。
- 第三方依赖需要手动下载。
- 依赖的传递性关系不容易维护。
- 依赖版本冲突不容易排查。
- 团队成员之间容易出现环境不一致。

## Maven 的核心思想

Maven 的重要理念是：

```text
Convention over configuration
约定优于配置
```

含义是：只要项目遵守 Maven 约定的目录结构、生命周期和配置方式，就可以少写很多重复配置。

Maven 的作用包括：

- 管理项目依赖。
- 下载和缓存第三方 jar 包。
- 规范项目目录结构。
- 执行编译、测试、打包、安装、部署等构建步骤。
- 支持插件扩展。
- 支持多模块项目的聚合和继承。

因此，Maven 不只是包管理工具，也是自动化构建工具。

## Maven 仓库

Maven 通过仓库管理依赖包。

常见仓库类型：

| 仓库类型 | 作用 |
| --- | --- |
| 本地仓库 | 存放当前机器已经下载过的依赖，默认在 `~/.m2/repository` |
| 中央仓库 | Maven 官方公共仓库，存放大量开源依赖 |
| 私服仓库 | 公司或团队内部搭建的仓库，用于缓存依赖或发布内部包 |

Maven 查找依赖时，通常会先查本地仓库；本地没有时，再去远程仓库下载。

本地仓库默认位置：

```text
~/.m2/repository
```

在 Windows 中通常类似：

```text
C:\Users\用户名\.m2\repository
```

## Maven 坐标

Maven 使用坐标唯一标识一个依赖。

核心坐标由三部分组成：

```text
groupId
artifactId
version
```

示例：

```xml
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <version>8.4.0</version>
</dependency>
```

含义：

| 坐标 | 说明 |
| --- | --- |
| `groupId` | 组织、公司或项目组标识 |
| `artifactId` | 具体模块或构件名称 |
| `version` | 版本号 |

Maven 会根据坐标到仓库中查找依赖。

## 语义化版本和 SNAPSHOT

常见版本格式：

```text
主版本号.次版本号.修订号
```

例如：

```text
1.4.2
```

一般含义：

- 主版本号：通常表示不兼容的大改动。
- 次版本号：通常表示向后兼容的新功能。
- 修订号：通常表示 bug 修复。

`SNAPSHOT` 表示快照版本，例如：

```text
1.0.0-SNAPSHOT
```

快照版本通常用于开发阶段，内容可能不断变化。正式发布版本一般不带 `SNAPSHOT`。

## Maven 标准项目结构

Maven 约定的常见项目结构：

```text
project
├── pom.xml
└── src
    ├── main
    │   ├── java
    │   └── resources
    └── test
        ├── java
        └── resources
```

各目录含义：

| 路径 | 作用 |
| --- | --- |
| `pom.xml` | Maven 项目配置文件 |
| `src/main/java` | 主程序 Java 源码 |
| `src/main/resources` | 主程序资源文件 |
| `src/test/java` | 测试 Java 源码 |
| `src/test/resources` | 测试资源文件 |
| `target` | 构建输出目录 |

遵守这套结构后，Maven 默认就知道源码、资源、测试和输出目录在哪里。

## pom.xml 的作用

`pom.xml` 是 Maven 项目的核心配置文件。

POM 全称是 `Project Object Model`，表示项目对象模型。

常见内容包括：

- 项目坐标。
- 依赖列表。
- 插件配置。
- 构建配置。
- 模块关系。
- 父子项目关系。

基础示例：

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 https://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.example</groupId>
    <artifactId>demo</artifactId>
    <version>1.0.0</version>

    <dependencies>
        <dependency>
            <groupId>com.mysql</groupId>
            <artifactId>mysql-connector-j</artifactId>
            <version>8.4.0</version>
        </dependency>
    </dependencies>
</project>
```

## 传递性依赖

传递性依赖表示：项目依赖的 jar 包还可以继续依赖其他 jar 包。

例如：

```text
当前项目
  -> A
      -> B
      -> C
```

项目只直接声明依赖 A，Maven 会自动解析并引入 B 和 C。

这能减少手动管理依赖的成本，但也会带来依赖冲突问题。

## 依赖冲突

Maven 的一个重要原则是：最终 classpath 中不应同时出现同一个 jar 的多个版本。

例如：

```text
项目依赖 A -> commons-xxx:1.0
项目依赖 B -> commons-xxx:2.0
```

如果两个版本都进入运行时 classpath，可能导致类加载混乱。

Maven 会根据规则选择其中一个版本，常见规则是：

```text
路径最近者优先
```

也就是依赖树中距离当前项目更近的版本胜出。

如果路径距离相同，通常先声明的依赖更容易胜出。

## 查看依赖树

排查 Maven 依赖问题时，常用命令是：

```bash
mvn dependency:tree
```

只查看某个依赖：

```bash
mvn dependency:tree -Dincludes=groupId:artifactId
```

例如：

```bash
mvn dependency:tree -Dincludes=com.fasterxml.jackson.core:jackson-databind
```

依赖树能帮助定位：

- 某个 jar 是被哪个依赖间接引入的。
- 同一个依赖出现了哪些版本。
- 冲突版本最终选择了哪个。

## 解决依赖冲突

常见处理方式：

### 1. 显式声明需要的版本

在当前项目中直接声明目标版本，让它成为更近的依赖。

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>some-library</artifactId>
    <version>2.0.0</version>
</dependency>
```

### 2. 排除传递性依赖

使用 `exclusions` 排除不需要的间接依赖。

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>a-library</artifactId>
    <version>1.0.0</version>
    <exclusions>
        <exclusion>
            <groupId>com.example</groupId>
            <artifactId>conflict-library</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

### 3. 使用 dependencyManagement 统一版本

在父工程或当前工程中统一声明依赖版本。

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>common-library</artifactId>
            <version>2.0.0</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

`dependencyManagement` 只管理版本，不会自动引入依赖。真正使用依赖时仍需要在 `dependencies` 中声明。

## 依赖 scope

`scope` 用来控制依赖在哪些阶段可用，以及是否会传递给其他项目。

常见 scope：

| scope | 说明 | 常见场景 |
| --- | --- | --- |
| `compile` | 默认作用域，编译、测试、运行都可用 | 普通业务依赖 |
| `test` | 只在测试阶段可用 | JUnit、Mockito |
| `provided` | 编译和测试可用，运行时由外部环境提供 | Servlet API |
| `runtime` | 编译时不需要，运行时需要 | JDBC 驱动 |
| `system` | 使用本地系统路径依赖，不推荐 | 特殊本地 jar |
| `import` | 用于导入 BOM，通常配合 `dependencyManagement` | Spring Boot 依赖管理 |

示例：测试依赖。

```xml
<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <version>5.10.2</version>
    <scope>test</scope>
</dependency>
```

示例：Servlet API。

```xml
<dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <version>6.0.0</version>
    <scope>provided</scope>
</dependency>
```

## 常见依赖异常

### ClassNotFoundException

运行时按类名加载类，但 classpath 中找不到这个类。

常见原因：

- 缺少依赖。
- 依赖没有进入运行时 classpath。
- 类名写错。

### NoClassDefFoundError

编译时类存在，但运行时找不到类定义。

常见原因：

- 编译环境和运行环境依赖不一致。
- 打包时漏掉依赖。
- 依赖 scope 设置不正确。

### AbstractMethodError

运行时调用了某个抽象方法或接口方法，但实际加载的类版本没有正确实现该方法。

常见原因：

- 编译时和运行时 jar 版本不一致。
- 依赖冲突导致加载了旧版本类。

### LinkageError

类链接阶段出现错误，常与二进制兼容性、重复类、版本冲突有关。

常见原因：

- 同一类存在多个版本。
- jar 包版本不兼容。
- 运行时加载到了错误版本。

## Maven 生命周期

Maven 构建过程由生命周期管理。

常见命令：

| 命令 | 作用 |
| --- | --- |
| `mvn clean` | 清理构建输出 |
| `mvn compile` | 编译主代码 |
| `mvn test` | 执行测试 |
| `mvn package` | 打包项目 |
| `mvn install` | 安装到本地仓库 |
| `mvn deploy` | 发布到远程仓库 |

Maven 生命周期有顺序性。

例如执行：

```bash
mvn package
```

Maven 会先执行它前面的阶段，例如编译和测试，然后再打包。

## Maven 插件

Maven 的很多功能由插件完成。

例如：

- 编译 Java 代码：`maven-compiler-plugin`
- 执行单元测试：`maven-surefire-plugin`
- 打包 jar：`maven-jar-plugin`
- 打包可执行应用：`spring-boot-maven-plugin`

插件可以绑定到生命周期阶段，在特定阶段自动执行。

示例：配置 Java 编译版本。

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.13.0</version>
            <configuration>
                <source>17</source>
                <target>17</target>
            </configuration>
        </plugin>
    </plugins>
</build>
```

## 聚合和继承

Maven 支持多模块项目。

### 聚合

聚合用于一次性构建多个模块。

```xml
<modules>
    <module>user-service</module>
    <module>order-service</module>
</modules>
```

在父目录执行：

```bash
mvn package
```

Maven 会按模块依赖关系构建多个子模块。

### 继承

继承用于让多个子模块共享父 POM 中的配置。

常见共享内容：

- 依赖版本。
- 插件版本。
- 编译配置。
- 仓库配置。

子模块中声明父 POM：

```xml
<parent>
    <groupId>com.example</groupId>
    <artifactId>demo-parent</artifactId>
    <version>1.0.0</version>
</parent>
```

## 真实项目中的 Maven 使用建议

实际项目中可以重点关注：

- 项目基本结构是否符合 Maven 约定。
- `pom.xml` 中依赖是否清晰。
- 是否使用 `dependencyManagement` 统一版本。
- 是否存在重复依赖或冲突依赖。
- 是否正确设置 `scope`。
- 是否能通过 `mvn dependency:tree` 解释依赖来源。
- 是否能通过 `mvn clean package` 完成完整构建。

常用排查命令：

```bash
mvn dependency:tree
mvn dependency:tree -Dverbose
mvn help:effective-pom
mvn clean package
```

`effective-pom` 可以查看继承、默认配置和插件合并后的最终 POM，有助于理解 Maven 实际使用了哪些配置。

## 容易混淆的点

`Classpath` 是 JVM 查找类的位置集合，不等同于 Java 包名。

包名决定类的命名空间，目录结构通常与包名对应。

jar 包是类和资源的打包结果，不等同于 Maven 依赖；Maven 依赖最终通常会下载为 jar 包。

Maven 坐标由 `groupId`、`artifactId`、`version` 组成，用于唯一定位依赖。

本地仓库用于缓存依赖，默认是 `~/.m2/repository`。

`dependencies` 会真正引入依赖，`dependencyManagement` 主要用于统一管理版本。

`compile` 是默认 scope；`test` 只用于测试；`provided` 表示运行环境提供；`runtime` 表示运行时才需要。

依赖冲突不一定会在编译阶段暴露，很多问题会在运行时以类加载异常的形式出现。

Maven 不只是下载 jar 包，还负责构建生命周期、测试、打包、插件执行和多模块管理。

## 总结

Java 包管理的核心是让 JVM 正确找到运行所需的类，并处理第三方类库之间的依赖关系。手动维护 classpath 和 jar 包容易带来版本冲突、依赖缺失和环境不一致。Maven 通过仓库、坐标、标准目录结构、传递性依赖管理和构建生命周期，显著降低了 Java 项目的依赖管理和构建复杂度。掌握 Maven 时，应重点理解坐标、仓库、scope、依赖树、冲突解决、生命周期和插件机制。
