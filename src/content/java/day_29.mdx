# MySQL 事务基础

> 事务用于保证一组数据库操作要么一起成功，要么一起失败。ACID、隔离级别和 MVCC 都围绕一个核心问题展开：在并发场景下，如何同时兼顾数据正确性和执行性能。

## 先看结论

事务最简单的定义：

```text
一组 SQL 要么全部成功，要么全部失败
```

比如转账：

```text
张三 -1000
李四 +1000
```

这两步必须放在同一个事务里，避免出现扣款成功但收款失败的数据不一致问题。

MySQL 事务需要掌握以下重点：

- `commit`：提交事务，让修改真正生效。
- `rollback`：回滚事务，把已经做的修改撤销。
- 默认情况下，MySQL 通常是自动提交的。
- 并发事务会带来脏读、不可重复读、幻读。
- 隔离级别越高，数据越安全，但并发性能越差。
- MySQL 默认隔离级别是 `Repeatable Read`，也就是可重复读。
- MVCC 是 MySQL 在读已提交、可重复读下提高并发能力的关键。

## 内容结构

先理解“事务是什么”和“ACID”，建立基础概念。

再重点学习“并发问题”和“隔离级别”，这是分析数据库行为时最常用的基础。

最后学习 MVCC。MVCC 的关键不是直接背规则，而是理解它如何通过 `Read View + Undo Log` 判断“当前事务能看到哪个版本的数据”。

## 事务是什么

事务 `Transaction` 是一组数据库操作。

这组操作有一个要求：要么全部执行成功，要么全部执行失败，不允许只成功一部分。

银行转账是最经典的例子：

```sql
update account set money = money - 1000 where name = '张三';
update account set money = money + 1000 where name = '李四';
```

如果第一条执行成功，第二条还没执行系统就崩了，就会出现钱凭空消失的问题。

所以这两条 SQL 应该放在同一个事务里：

```sql
begin;

update account set money = money - 1000 where name = '张三';
update account set money = money + 1000 where name = '李四';

commit;
```

如果中间出问题：

```sql
rollback;
```

事务不是为了保证 SQL 一定成功，而是为了在失败时能够回到事务执行前的状态。

## 自动提交和手动事务

MySQL 默认一般是自动提交的。

也就是说，如果直接执行一条 `update`：

```sql
update student set age = 18 where id = 1;
```

它会作为一个事务自动提交。所以另一个连接马上就能看到修改后的结果。

如果想手动控制事务，就要显式开启：

```sql
begin;

update student set age = 20 where id = 1;

commit;
```

提交前，同一个事务里能看到本事务已经修改的数据；其他事务不一定能看到，要看隔离级别。

回滚演示：

```sql
begin;

update student set age = 22 where id = 1;

rollback;
```

回滚之后，`age = 22` 这次修改就被撤销了。

演示多个事务时，要用两个数据库连接。同一个连接里开来开去，本质上还是同一个会话，效果容易看乱。

## ACID 四大特性

事务的四大特性叫 `ACID`。

### A：原子性 Atomicity

一个事务是一个整体，要么全部成功，要么全部失败回滚。

转账里扣钱和加钱必须一起成功，不能只执行一半。

### C：一致性 Consistency

事务执行前后，数据库都应该处于正确状态。

比如金额不能凭空增加或消失，外键、唯一约束这些规则也不能被破坏。

### I：隔离性 Isolation

多个事务并发执行时，彼此之间应该尽量互不干扰。

但“完全不干扰”成本很高，所以数据库提供了不同隔离级别。

### D：持久性 Durability

事务提交之后，数据要真正保存下来。

哪怕系统崩溃或断电，也不应该把已经提交的数据弄丢。

## 并发事务会有什么问题

事务并发就是多个事务同时执行。

并发的好处是性能高，但问题是读写之间可能互相影响。

主要有三个问题：

```text
脏读 -> 读到了其他事务还没提交的数据
不可重复读 -> 同一事务里两次读同一行，结果变了
幻读 -> 同一事务里两次按条件查询，结果集行数变了
```

## 脏读

脏读：一个事务读到了另一个事务还没提交的数据。

例子：

```text
事务 A：update student set age = 20 where id = 1; 但还没 commit
事务 B：select age from student where id = 1; 读到了 20
事务 A：rollback
```

这时事务 B 读到的 `20` 就是脏数据，因为事务 A 最后回滚了。

要点：未提交的数据被其他事务读到，就是脏读。

## 不可重复读

不可重复读：同一个事务里，两次读取同一条数据，值不一样。

例子：

```text
事务 A 第一次读 id = 1，age = 18
事务 B 修改 id = 1，age = 20，并提交
事务 A 第二次读 id = 1，age = 20
```

事务 A 明明没结束，但两次读同一行结果不同，这就是不可重复读。

要点：不可重复读关注的是“同一行数据的值变了”。

## 幻读

幻读：同一个事务里，两次按同样条件查询，查出来的行集不一样。

例子：

```text
事务 A 查询 age >= 18 的学生，有 2 条
事务 B 插入一个 age = 20 的学生，并提交
事务 A 再查 age >= 18 的学生，变成 3 条
```

事务 A 像是突然“看见了新行”，所以叫幻读。

区分：不可重复读更偏“同一行变了”，幻读更偏“多了/少了行”。

## 事务隔离级别

隔离级别从低到高：

```text
Read Uncommitted  读未提交
Read Committed    读已提交
Repeatable Read   可重复读
Serializable      串行化
```

越往下，隔离越强，问题越少，但并发性能也越差。

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 备注 |
| --- | --- | --- | --- | --- |
| 读未提交 | 可能 | 可能 | 可能 | 最低级别，基本不常用 |
| 读已提交 | 避免 | 可能 | 可能 | 只能读到其他事务已提交的数据 |
| 可重复读 | 避免 | 避免 | 可能 | MySQL 默认级别 |
| 串行化 | 避免 | 避免 | 避免 | 最严格，但性能差 |

查看当前隔离级别：

```sql
select @@transaction_isolation;
```

设置隔离级别：

```sql
set session transaction isolation level read uncommitted;
set session transaction isolation level read committed;
set session transaction isolation level repeatable read;
set session transaction isolation level serializable;
```

也可以设置全局：

```sql
set global transaction isolation level repeatable read;
```

注意：设置完隔离级别后，最好重新打开连接，让设置生效。

## 四个隔离级别

### 读未提交 Read Uncommitted

这个级别最低，允许读到其他事务还没提交的数据。

所以它可能出现：

- 脏读
- 不可重复读
- 幻读

这个级别隔离能力较弱，实际开发里很少会主动选择。

### 读已提交 Read Committed

只能读其他事务已经提交的数据，所以可以避免脏读。

但是同一个事务里，如果别的事务提交了修改，当前事务下次再读可能就读到新值。

所以它还可能出现：

- 不可重复读
- 幻读

### 可重复读 Repeatable Read

MySQL 默认隔离级别。

它保证同一个事务里，多次普通查询同一条数据，结果保持一致。

所以它可以避免：

- 脏读
- 不可重复读

但在某些场景下仍然可能出现幻读。

关键点：可重复读的快照不是在 `begin` 时生成的，而是在第一次执行 SQL 时生成的。

### 串行化 Serializable

最高隔离级别。

事务基本排队执行，读也会加锁，直到事务结束才释放。

它可以避免：

- 脏读 
- 不可重复读
- 幻读

但代价是并发性能差，还更容易出现锁等待甚至死锁。

所以一般没有特殊要求，通常不会使用串行化。

## 快照读和当前读

分析 MySQL 可重复读下的幻读，要先区分两种读。

### 快照读 snapshot read

普通 `select` 通常是快照读：

```sql
select * from student where age >= 18;
```

快照读读的是某个历史一致性视图，不一定是最新数据。

在可重复读级别下，普通 `select` 会尽量保证同一个事务里多次读取结果一致。

### 当前读 current read

`update`、`delete`、`insert`，以及带锁查询通常是当前读。

例如：

```sql
update student set age = age + 1 where age >= 18;
```

当前读读的是最新已经提交的数据，并且可能加锁。

所以可重复读下会出现一种容易混淆的情况：

```text
普通 select 读的是快照
update 用的是当前读
事务内可以读到本事务刚刚 update 的数据
```

这就是为什么可重复读有时看起来解决了幻读，有时又会出现幻读。

## MVCC 是什么

`MVCC` 全称是 `Multi-Version Concurrency Control`，多版本并发控制。

可以理解为：

```text
同一行数据保留多个历史版本
读的时候根据规则判断当前事务能看到哪个版本
```

这样读操作不用总是加锁，读写并发能力就会好很多。

MySQL 的 `Read Committed` 和 `Repeatable Read` 都是基于 MVCC 实现的。

## MVCC 需要记住的几个东西

### 事务 ID

事务开启时，会拿到一个自增长的事务 ID。

事务 ID 可以判断事务开始的先后顺序，但不能简单判断提交顺序。

因为事务 ID 小的事务，也可能比事务 ID 大的事务更晚提交。

### 隐式字段

InnoDB 每行记录里有隐藏字段，重点记两个：

- `trx_id`：最后修改这行数据的事务 ID。
- `roll_pointer`：回滚指针，指向 Undo Log 里的旧版本。

如果表没有主键和非空唯一键，InnoDB 还会额外生成一个隐藏的 `row_id`。

### Undo Log

`Undo Log` 是回滚日志，记录数据被修改前的信息。

它有两个用途：

- 事务回滚。
- MVCC 快照读。

### 版本链

同一行数据被多次修改，就会产生多个版本。

这些版本通过 `roll_pointer` 串起来，就形成版本链。

可以想成：

```text
最新版本 -> 上一个版本 -> 再上一个版本 -> ...
```

查询时，如果最新版本对当前事务不可见，就顺着版本链往旧版本找。

## Read View 是什么

`Read View` 是事务执行查询时生成的读视图。

它的作用是做可见性判断：

```text
当前事务到底能不能看见某个版本的数据？
```

Read View 里几个关键属性：

- `creator_trx_id`：创建这个 Read View 的事务 ID。
- `m_ids`：生成 Read View 时，系统中还活跃的事务 ID 列表。
- `min_limit_id`：活跃事务 ID 里的最小值。
- `max_limit_id`：生成 Read View 时，系统将要分配给下一个事务的 ID。

简化理解：

```text
m_ids 里面的是当时还没提交的事务
min_limit_id 和 max_limit_id 用来划范围
creator_trx_id 用来判断是不是当前事务修改的数据
```

## Read View 可见性规则

判断一行版本能不能看，主要看这个版本的 `trx_id`。

### 1. trx_id < min_limit_id

说明这个版本在 Read View 生成前就已经提交了。

所以可见。

### 2. trx_id >= max_limit_id

说明这个版本是在 Read View 生成后才出现的。

所以不可见。

### 3. `min_limit_id <= trx_id < max_limit_id`

这个范围比较麻烦，要看 `m_ids`。

如果 `m_ids` 不包含这个 `trx_id`：

```text
说明生成 Read View 时，这个事务已经提交了
所以可见
```

如果 `m_ids` 包含这个 `trx_id`，并且 `trx_id == creator_trx_id`：

```text
说明这是当前事务修改出来的数据
所以可见
```

如果 `m_ids` 包含这个 `trx_id`，并且 `trx_id != creator_trx_id`：

```text
说明生成 Read View 时，其他事务还没提交
所以不可见
```

一句话版：

```text
早就提交的能看
未来才出现的不能看
当前事务修改的版本可见
当时其他事务还没提交的版本不可见
```

## MVCC 查询一条数据的大概过程

InnoDB 做 MVCC 查询时，大概是这样：

```text
1. 获取当前事务 ID
2. 获取 Read View
3. 读取数据的最新版本
4. 用 Read View 判断这个版本是否可见
5. 如果不可见，就沿着 Undo Log 版本链找旧版本
6. 找到第一个可见版本后返回
```

所以 MVCC 的核心组合是：

```text
Read View + Undo Log
```

`Undo Log` 负责提供历史版本。

`Read View` 负责判断哪个版本能看。

## 读已提交和可重复读的 MVCC 区别

这两个隔离级别都用 MVCC，区别在于生成 Read View 的时机。

### 读已提交

每次读都会重新生成一个新的 Read View。

所以如果别的事务提交了新数据，当前事务下一次读就可能读到新结果。

这就是为什么读已提交会出现不可重复读。

### 可重复读

第一次读生成 Read View，后面普通查询继续复用这个 Read View。

所以同一个事务里，多次普通 `select` 看到的结果可以保持一致。

这就是为什么可重复读能解决不可重复读。

对比方式：

```text
Read Committed：每次 select 都生成新的 Read View
Repeatable Read：第一次 select 生成 Read View，后续普通查询复用该 Read View
```

## 可重复读下为什么还可能幻读

课件里强调了一个细节：可重复读下，普通 `select` 是快照读，`update` 是当前读。

所以可能出现这种情况：

```text
事务 A 第一次 select，只看到张三、李四
事务 B 插入王五并提交
事务 A 第二次普通 select，还是只看到张三、李四
事务 A 执行 update，update 是当前读，会处理到王五
事务 A 再 select，因为事务 A 已经更新过王五，于是能看到王五
```

结果就是：事务 A 之前没看到王五，后面又看到了王五，像是出现了幻读。

这里容易混淆，关键点是：

```text
普通 select 看快照，update 看最新；当前事务 update 过的数据，当前事务后续可以读取到。
```

## 容易混的点

`commit` 是提交，`rollback` 是回滚。

MySQL 默认一般自动提交，所以一条独立 SQL 执行后，其他连接通常能马上看到结果。

事务隔离级别越高，并发问题越少，但性能越低。

脏读：读到其他事务没提交的数据。

不可重复读：同一事务里读同一行，两次值不一样。

幻读：同一事务里按同样条件查，两次结果集行数不一样。

MySQL 默认隔离级别是可重复读。

可重复读的快照不是 `begin` 时生成的，而是第一次执行 SQL 时生成的。

MVCC 的关键是 `Read View + Undo Log`。

`Read Committed` 每次查询都会生成新的 Read View。

`Repeatable Read` 第一次查询生成 Read View，后面普通查询复用。

## 一句话总结

事务解决的是“一组操作要么全成要么全败”的问题；隔离级别解决的是“多个事务一起跑时互相影响到什么程度”的问题；MVCC 则是 MySQL 为了在保证隔离性的同时提高读写并发，用多版本数据和 Read View 可见性规则做出来的一套机制。
