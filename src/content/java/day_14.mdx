## 哈希表基本原理
- 数组通过下标访问数据的一种拓展
- 核心：利用哈希函数将键值映射到数组上
  比如"abc"根据ascii码转成数字取模获得数组下标。当不同的key计算出的数字相同时也就是哈希冲突

### 哈希函数
- 哈希函数式用来将一个字符串（或任何其他类型）转化为小于哈希表大小且大于等于0的整数
一个好的哈希函数
1. 可以尽可能少的产生哈希冲突
2. 算的快

一种广泛使用的哈希函数算法是使用数值33（Times33算法）,java里面是改成了31
```java
public int hashCode(char[] key, int hashSize) {
    long result = 0;
    for (int i = 0;i < key.length;i++) {
        result = (result * 33 + (int)(key[i])) % hashSize
    }
    return (int)result
}
```

### 哈希冲突
- 表面原因：有一些key会map到相同index上
- 本质：无限空间往有限空间映射

如何解决哈希冲突
- 设计好的hash函数
- 改变哈希索引
  - Open hashing
  - Closed hashing
- 扩容

### Closed hashing 闭散列
开放定址法
线性探测
``` 
hash = (hash(key) + i) % hashSize
i = 0,1,2,3....
```
也就是取模探测到相同key值的时候加一个i，直到整个哈希表被填满
设置和查找是同样的操作。当查找的时候遍历到空值时，就说明要查找的key不在散列表中。
删除的时候要做一个delete标记，这样查找的时候不影响。

问题是空间小，并且遍历时间复杂度高。删除等操作复杂。

无论是线性探测/二次探测/双重散列都有空间问题，导致潜在冲突增加。

### Open Hash 开散列

- 每个bucket对应一条链表，哈希值相同的元素直接连接在对应链表中
- 链表的头结点存储在哈希表中

缺点：复杂、不能直接访问元素，要间接访问指针。

极端情况所有key都在一个bucket中，链表非常长，导致变成了单纯的链表查找。

java 7底层数据结构式数组+链表  8 是数组+链表+红黑树。

### 扩容：重哈希rehashing
- 哈希表容量的大小在一开始是不确定的，在需要的时候，可以对底层数组进行扩容
- 一种简单策略：如果哈希表存储元素太多，将哈希表容量扩大一倍，并将所有的key的哈希值重新计算映射到新的bucket上。
