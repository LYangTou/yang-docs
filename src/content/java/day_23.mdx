## 2. Cookie 的定义

Cookie 是用户浏览网站时，由 Web 服务器创建，并由浏览器保存到用户计算机或其他设备上的小文本文件。

可以从四个角度理解 Cookie：

| 角度 | 说明 |
|---|---|
| 是什么 | 小文本文件 |
| 什么时候使用 | 浏览网站时 |
| 谁创建 | 服务端创建 |
| 存在哪里 | 客户端浏览器 |

## 3. Cookie 的作用

Cookie 的作用是让 Web 服务器能够在浏览器中存储状态信息，从而解决 HTTP 协议无状态的问题。

### HTTP 无状态的含义

HTTP 协议是无状态的，意思是：

- 每个 HTTP 请求都是独立的。
- 单个请求中包含处理该请求所需的全部信息。
- 服务端默认不会记住之前的请求信息。
- 客户端发送请求前建立连接，服务端响应后连接关闭。
- 下一次请求需要重新建立连接，上一轮请求的信息不会自动保留。

### 无状态带来的问题

无状态会给登录验证带来困难。

例如：

1. 用户先发送登录请求。
2. 用户再发送查询房间等其他请求。
3. 因为服务端默认不记住上一次登录请求，所以后续请求无法直接判断是不是之前登录过的用户。

### 为什么 HTTP 要设计成无状态

HTTP 设计成无状态有几个好处：

- 简化服务器管理，服务器不需要为每个客户端保存大量状态信息。
- 减轻服务器负担，提高系统效率和稳定性。
- 便于 Web 应用水平扩展。
- 请求和响应之间的关系更清晰，协议更容易维护和演进。

### 如何解决登录验证问题

常见方案：

- 每次请求都让用户重新输入登录信息，体验很差。
- 将用户信息以 Cookie 的形式保存到客户端，后续请求自动携带 Cookie。

结论：

HTTP 协议本身是无状态的，但可以通过 Cookie、Session 等技术让 Web 应用表现出“有状态”。

## 4. Cookie 的属性

### 4.1 Name

Cookie 的名称，用于识别 Cookie。

服务端创建 Cookie 并返回给浏览器时，响应头中会包含 `Set-Cookie` 字段，指示浏览器保存 Cookie。

```java
Cookie cookie = new Cookie("name", "afaf")
```

### 4.2 Value

Cookie 的值，用于存储保存在客户端的数据。

### 4.3 Domain

Cookie 所属的域名，用于限制 Cookie 的作用域。

浏览器只会向指定域名发送 Cookie。

如果不配置 `Domain`，默认是当前域名。

### 4.4 Path

Cookie 所属的路径，用于限制 Cookie 的作用路径。

浏览器只会向指定路径发送 Cookie。

如果不配置 `Path`，默认路径通常是当前请求路径的上一层地址。

例如访问：

```text
http://localhost:8080/hotel/cookie/create
```

如果创建 Cookie 时没有指定 `Path`，那么默认路径可能是：

```text
/hotel/cookie
```

此时：

- 访问 `/hotel/cookie/query` 会发送该 Cookie。
- 访问 `/hotel/queryRoom` 不会发送该 Cookie。

如果希望当前项目下所有路径都能携带 Cookie，通常可以设置：

```java
cookie.setPath(request.getContextPath());
```

### 4.5 Max-Age

`Max-Age` 表示 Cookie 的存活时间，单位是秒。

| 值 | 含义 |
|---|---|
| 负数 | Cookie 只保存在浏览器内存中，关闭浏览器后消失 |
| 正数 | Cookie 保存在客户端电脑中，指定秒数后失效 |
| 0 | 删除 Cookie |

说明：

- `Max-Age` 默认为 `-1`。
- `Max-Age` 为负数时，这类 Cookie 叫做会话 Cookie。
- `Max-Age` 为正数时，这类 Cookie 叫做持久性 Cookie。
- `Max-Age = 0` 常用于退出登录。

### 4.6 Expires

`Expires` 表示 Cookie 的失效时间，是一个具体的日期时间。

当 `Max-Age` 为正数时，`Expires` 属性才有意义。

### 4.7 Secure

`Secure` 用于控制 Cookie 是否只通过 HTTPS 发送。

默认值是 `false`。

如果 Cookie 在 HTTPS 下创建，并且 `Secure = true`：

- 后续使用 HTTPS 访问时，Cookie 会发送到服务器。
- 后续使用 HTTP 访问时，Cookie 不会发送到服务器。

### 4.8 HttpOnly

`HttpOnly` 用于限制 JavaScript 是否可以访问 Cookie。

默认值是 `false`。

如果设置为 `true`：

- Cookie 不能被 JavaScript 访问。
- Cookie 不能被 JavaScript 修改。
- Cookie 的安全性更高。

示例：

- `username` 没有设置 `HttpOnly`，可以在浏览器 Console 中通过 JavaScript 访问和修改，不安全。
- `age` 设置了 `HttpOnly`，不能通过 JavaScript 访问和修改，安全性更高。

## 5. 使用 Cookie 实现 30 天免登录

### 5.1 基本需求

功能流程：

1. 用户注册。
2. 用户登录。
3. 登录成功后，在 Cookie 中保存用户信息。
4. 用户查询房间时，服务端根据请求中是否携带指定 Cookie 判断是否登录过。
5. Cookie 的过期时间设置为 30 天。

### 5.2 基本实现思路

登录成功后保存 Cookie：

```text
客户端提交用户名和密码
        |
        v
服务端验证用户名和密码
        |
        v
验证通过后创建用户信息 Cookie
        |
        v
响应给客户端，浏览器保存 Cookie
```

查询房间时验证 Cookie：

```text
客户端发送查询房间请求
        |
        v
服务端检查用户是否已登录
        |
        |-- 未登录：跳转到登录页面
        |
        |-- 已登录：从 Cookie 中获取用户信息，执行业务逻辑
```

### 5.3 密码修改场景

如果只在 Cookie 中保存 `username`，会存在问题：

- 用户修改密码后，只要 Cookie 中还带着旧的 `username`，服务端仍可能误认为用户已登录。

更合理的方案：

- 登录成功后，在 Cookie 中保存 `username` 和 `password`。
- 密码建议加密后再保存。
- 后续请求中，根据 Cookie 中的 `username`、`password` 以及数据库中的密码是否一致判断是否登录过。

密码修改后：

```text
客户端提交用户名、旧密码、新密码
        |
        v
服务端验证用户名和旧密码
        |
        v
验证通过后更新为新密码
```

再次查询房间时：

```text
服务端从 Cookie 中获取用户名和密码
        |
        v
校验用户名和密码是否正确
        |
        v
校验通过后执行业务逻辑
```

## 6. Session 的定义

Session，即会话，指客户端与服务器之间一段持续的交互过程。

通常从用户打开浏览器访问网站开始，到用户关闭浏览器结束。

在这个交互过程中：

- 服务端为每个客户端维护一个唯一标识符，即 `session id`。
- 服务端通过 `session id` 识别不同客户端的会话状态。
- 会话数据保存在服务端。
- 客户端只保存 `session id`。

可以从四个角度理解 Session：

| 角度 | 说明 |
|---|---|
| 是什么 | 客户端与服务端之间一段持续交互的过程 |
| 生命周期 | 打开网站到关闭浏览器 |
| 谁创建 | 服务端创建 |
| 存在哪里 | 服务端，客户端只保存 `session id` |

## 7. Session 的作用与使用

Session 的作用是维护用户的会话状态，解决 HTTP 协议无状态的问题。

常见用途：

- 用户登录状态
- 购物车
- 临时业务数据

### 7.1 获取 Session
从`HttpServlet`的req中取

```java
public HttpSession getSession();
```

作用：

- 获取当前 Session。
- 如果 Session 不存在，则创建一个新的 Session。

```java
public HttpSession getSession(boolean create);
```

作用：

- 获取当前 Session。
- 如果 Session 不存在，是否创建由参数 `create` 决定。

### 7.2 设置 Session 属性

```java
public void setAttribute(String name, Object value);
```

作用：

- 向 Session 中保存数据。

### 7.3 获取 Session 属性

```java
public Object getAttribute(String name);
```

作用：

- 从 Session 中根据属性名获取数据。

## 8. Session 的实现原理

Session 的实现基于 Cookie。

核心逻辑：

1. 客户端第一次发送请求。
2. 服务端创建 Session，并生成唯一的 `session id`。
3. 服务端通过响应头 `Set-Cookie` 将 `session id` 返回给浏览器。
4. 浏览器保存包含 `session id` 的 Cookie。
5. 后续请求会自动携带该 Cookie。
6. 服务端根据 `session id` 找到对应的 Session 数据。

流程：

```text
客户端发送请求
        |
        v
服务端创建 Session，生成 session id
        |
        v
响应头返回 Set-Cookie: sessionId=xxx
        |
        v
浏览器保存 sessionId
        |
        v
后续请求自动携带 sessionId
        |
        v
服务端根据 sessionId 获取对应 Session
```

Session 在服务端可以理解为嵌套 Map 结构：

```java
Map<String, Map<String, Object>> sessionMap;
```

其中：

- 外层 Map 的 key 是 `session id`。
- 内层 Map 的 key 是 `attributeName`。
- 内层 Map 的 value 是 `attributeValue`。

## 9. Session 的过期机制

### 9.1 为什么 Session 需要过期机制

服务端需要为每个客户端维护一个 Session。

如果客户端数量越来越多，服务端维护的 Session 数据也会越来越多。

Session 数据保存在 Web 服务器内存中，如果不清理，可能会造成：

- 内存占用过高。
- 服务器资源浪费。
- 大量长时间不活跃的 Session 残留。

例如，用户打开网页后长时间离开电脑或断开网络连接，如果 Session 一直保留，就会浪费服务器资源。

### 9.2 方式一：设置过期时间

可以通过 `setMaxInactiveInterval()` 设置 Session 的过期时间，单位为秒。

```java
session.setMaxInactiveInterval(1800);
```

说明：

- 设置的是 Session 的最大不活跃时间。
- 如果在这段时间内访问了该 Session，Session 会续期。
- 如果超过这段时间没有访问，Session 会过期。

### 9.3 方式二：手动过期

可以通过 `invalidate()` 强制让 Session 失效。

```java
session.invalidate();
```

说明：

- 手动过期是一种特殊操作。
- 一般不建议随意使用。
- 常见场景是用户退出登录。

## 10. 使用 Session 实现购物车

购物车适合使用 Session 保存，因为购物车数据属于用户会话状态。

基本需求：

- 加入购物车：向购物车中添加商品。
- 查看购物车：查询当前购物车中的商品。

基本思路：

```text
用户点击加入购物车
        |
        v
服务端获取当前用户 Session
        |
        v
从 Session 中获取购物车对象
        |
        |-- 如果购物车不存在，则创建购物车
        |
        v
将商品加入购物车
        |
        v
把购物车保存回 Session
```

查看购物车：

```text
用户访问购物车页面
        |
        v
服务端获取当前 Session
        |
        v
从 Session 中取出购物车
        |
        v
返回购物车商品列表
```

## 11. Cookie 和 Session 的区别

Cookie 和 Session 都可以用来跟踪用户访问状态、保存用户信息，但它们有明显区别。

| 对比项 | Cookie | Session |
|---|---|---|
| 存储位置 | 客户端浏览器 | 服务端内存 |
| 生命周期 | 可以设置保存时间，也可以浏览器关闭后失效 | 通常浏览器关闭后失效，也可以由服务端过期策略控制 |
| 数据类型 | 只能存储字符串 | 可以存储任意类型对象 |
| 存储容量 | 单个 Cookie 通常不超过 4KB | 理论上没有 Cookie 那样的小容量限制，但受服务器内存限制 |
| 安全性 | 存在客户端，安全性较低 | 存在服务端，安全性较高 |
| 典型用途 | 少量非敏感数据、用户状态标识 | 登录状态、购物车、敏感或较复杂数据 |

使用建议：

- 如果只需要保存少量数据，或跟踪简单用户状态，可以使用 Cookie。
- 如果需要保存大量数据、对象数据或敏感信息，建议使用 Session。

## 12. 核心总结

- Cookie 是服务端创建、客户端保存的小文本文件。
- Cookie 可以让浏览器在后续请求中携带状态信息。
- Cookie 主要用于解决 HTTP 无状态带来的状态保持问题。
- Cookie 的重要属性包括 `Name`、`Value`、`Domain`、`Path`、`Max-Age`、`Expires`、`Secure`、`HttpOnly`。
- Session 是客户端与服务端之间的一段持续会话。
- Session 数据保存在服务端，客户端只保存 `session id`。
- Session 的实现依赖 Cookie 传递 `session id`。
- Session 可以通过 `setAttribute()` 和 `getAttribute()` 保存、获取数据。
- Session 可以通过 `setMaxInactiveInterval()` 设置过期时间。
- Cookie 适合保存少量非敏感数据，Session 更适合保存敏感或复杂数据。
