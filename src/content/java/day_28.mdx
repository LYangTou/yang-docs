## 2. 什么是索引？

索引，英文是 `index`，在 MySQL 中也常叫做“键（key）”。

它是一种用于快速找到记录的数据结构。MySQL 的存储引擎可以根据索引快速定位到对应的数据行。

可以类比一本书的目录：

```text
没有目录：从第一页开始逐页翻找
有目录：先根据主题找到页码，再直接翻到目标页
```

数据库也是类似：

```text
没有索引：可能需要扫描很多行，甚至全表扫描
有索引：先通过索引定位，再找到对应数据
```

### 2.1 索引的作用

PPT 中提到索引主要有以下作用：

| 作用 | 说明 |
| --- | --- |
| 提高查询效率 | 减少扫描次数，加快查询 |
| 保证唯一性和完整性 | 唯一索引、主键索引可以限制重复数据 |
| 加速排序和分组 | 在某些场景下可以减少排序、分组成本 |
| 减少锁竞争 | 查询更快、扫描范围更小，有助于降低并发下的锁竞争 |

### 2.2 索引不是越多越好

索引会提升查询效率，但也有成本。

建立和维护索引会消耗：

- 磁盘空间
- 插入数据的成本
- 更新数据的成本
- 删除数据的成本

原因是：表数据变化时，相关索引也要同步变化。

所以建索引时要考虑：

- 这个列是否经常用于查询条件
- 这个列是否经常用于排序或分组
- 这个列区分度是否足够高
- 是否已经有类似的联合索引可以覆盖
- 索引数量是否过多

> 补充：索引的价值是“用写入和存储成本，换取查询性能”。

## 4. 哈希索引

哈希索引，英文是 `Hash Index`。

它基于哈希表实现，会把索引列的值通过哈希函数计算成一个哈希值，然后将哈希值和数据地址关联起来。

可以类比 Java 中的 `HashMap`：

```text
key -> hash(key) -> bucket -> 找到数据
```

### 4.1 哈希索引的优势

哈希索引非常适合等值查询。

例如：

```sql
select * from user where id = 1001;
```

如果哈希冲突少，就可以通过哈希值快速定位数据，速度很快。

### 4.2 哈希索引的劣势

PPT 中列出了几个重要限制：

| 限制 | 原因 |
| --- | --- |
| 只适合等值查询 | 哈希值不保留原始大小关系 |
| 不支持范围查询 | 无法高效处理 `>`、`<`、`between` |
| 无法用于排序 | 哈希表不是按索引值顺序存储 |
| 不支持部分索引列匹配 | 联合哈希索引通常用全部列计算哈希值 |
| 哈希冲突会增加维护成本 | 不同值可能得到相同哈希值 |

例如联合哈希索引 `(A, B)`，查询条件只有 `A` 时通常无法利用这个哈希索引，因为哈希值是基于全部索引列计算出来的。

## 5. 树结构与数据库索引

### 5.1 什么是树？

树是一种非线性数据结构，由节点和节点之间的连接组成。

节点中可以保存：

- 数据元素
- 指向其他节点的指针

### 5.2 平衡二叉搜索树

PPT 中先介绍了平衡二叉搜索树。

平衡二叉搜索树满足：

- 左子树所有节点的值都小于根节点
- 右子树所有节点的值都大于根节点
- 左右子树也分别是二叉搜索树
- 每个节点的左右子树高度差不超过 1

例如查找值 `7`：

```text
根节点 10：7 < 10，往左找
节点 6：7 > 6，往右找
节点 7：7 = 7，查找完成
```

平衡二叉搜索树可以支持较快的查找、插入、删除。

### 5.3 为什么二叉树不适合做数据库索引？

虽然二叉树查找效率不错，但它不适合作为数据库索引的主要结构。

核心原因：数据库索引通常存储在磁盘上，而磁盘 IO 很慢。

PPT 中强调了几个点：

1. 索引文件可能很大，无法一次全部加载到内存
2. 查询时往往需要从磁盘读取多个磁盘页
3. 二叉树每个节点只有两个分叉，数据多时树高会比较高
4. 树越高，查询需要访问的节点越多，也就意味着更多磁盘 IO
5. 二叉树不能很好利用磁盘预读

### 5.4 磁盘预读与局部性原理

磁盘读取通常不是只读一个字节，而是一次读取一整块连续数据到内存。

这叫磁盘预读。

它的理论基础是局部性原理：

```text
当一个数据被使用时，它附近的数据也很可能马上被使用。
```

但二叉树中逻辑上相近的节点，在物理磁盘上不一定相邻。

例如逻辑上 `10` 的左子节点是 `6`，但它们在磁盘文件中可能距离很远。读取 `10` 所在磁盘页时，不一定能把 `6` 一起读进来，所以查找 `6` 时又需要一次磁盘 IO。

## 6. B 树索引

B 树，英文是 `B-Tree`，有时也写作 `B-树`，不要读成“B 减树”。

它是一种多路平衡搜索树。

### 6.1 B 树的特点

PPT 中总结了 B 树的特点：

- 每个节点包含多个关键字和对应的数据
- 关键字按从小到大排列
- 非叶子节点包含多个指向子节点的指针
- 所有叶子节点位于同一层
- 每个节点可以有多个分叉

可以理解为：

```text
二叉树：一个节点最多 2 个孩子
B 树：一个节点可以有很多个孩子
```

### 6.2 B 树为什么比二叉树更适合数据库？

B 树的关键优势是降低树高。

因为每个节点可以保存多个关键字和多个指针，所以同样数量的数据，B 树比二叉树矮得多。

树高降低意味着：

```text
访问节点次数减少
磁盘 IO 次数减少
查询效率提高
```

PPT 中的例子是：查找关键字 `12` 时，二叉树需要读 5 次磁盘，而 B 树只需要读 3 次。

## 7. B+ 树索引

B+ 树，英文是 `B+Tree`，读作“B 加树”。

它是 B 树的变体，也是多路搜索树。

### 7.1 B+ 树的特点

PPT 中提到：

- 非叶子节点只存储关键字，不存储完整数据
- 非叶子节点保存指向子节点的指针
- 所有数据都存储在叶子节点
- 叶子节点之间有指针相连，形成有序链表

可以粗略理解为：

```text
非叶子节点：只负责导航
叶子节点：真正保存数据或数据地址
叶子节点之间：按顺序串起来
```

### 7.2 B+ 树相比 B 树的优势

PPT 中总结了两个核心优势。

**1. 查询效率更高**

B+ 树的非叶子节点不保存完整数据，只保存关键字和指针。

这意味着同一个磁盘页可以放下更多关键字。

结果是：

```text
单个节点能分出更多路
树的高度更矮
磁盘 IO 更少
查询更快
```

**2. 更适合范围查询**

B+ 树的叶子节点之间通过指针形成有序链表。

范围查询时，只需要：

```text
先找到范围左端点
再沿着叶子节点链表向右扫描
直到范围右端点
```

而 B 树做范围查询通常需要中序遍历，复杂度更高。

数据库中经常使用范围查询，例如：

```sql
select * from employees
where hire_date between '1989-01-01' and '1989-12-31';
```

因此 MySQL InnoDB 使用 B+ 树作为核心索引结构。

## 8. 索引的分类

PPT 中从多个角度对索引做了分类。

### 8.1 按功能分类

| 类型 | 说明 |
| --- | --- |
| 主键索引 `Primary Key` | 唯一标识表中记录，不允许重复，不允许为 `NULL`，一张表只能有一个 |
| 唯一索引 `Unique Index` | 索引值不允许重复，通常允许为 `NULL`，一张表可以有多个 |
| 普通索引 / 非唯一索引 | 索引值允许重复，也允许为 `NULL` |

注意：

```text
主键索引 = 唯一 + 非空 + 一张表只能一个
唯一索引 = 唯一约束，但通常允许 NULL
普通索引 = 只提升查询，不限制唯一性
```

### 8.2 按列数分类

| 类型 | 说明 |
| --- | --- |
| 单列索引 | 索引只包含一个列 |
| 联合索引 / 组合索引 | 索引包含多个列 |

例如：

```sql
-- 单列索引
alter table student add index idx_province (province);

-- 联合索引
alter table student add index idx_name_province (name, province);
```

### 8.3 按物理存储分类

PPT 中重点讲了聚簇索引和非聚簇索引。

| 类型 | 说明 |
| --- | --- |
| 聚簇索引 `Clustered Index` | 数据和索引放在同一个 B+ 树结构中，找到索引也就找到了数据 |
| 非聚簇索引 | 数据和索引分开存储，叶子节点存放的不是完整数据行 |

在 InnoDB 中：

```text
主键索引是聚簇索引
辅助索引 / 二级索引是非聚簇索引
```

### 8.4 聚簇索引的优缺点

**优点：按主键查询很快**

因为聚簇索引把索引和数据保存在同一棵 B+ 树中，找到索引就找到数据，不需要额外查找。

**缺点：插入速度依赖主键顺序**

如果按照主键顺序插入，通常只需要插入到最后一条记录后面，效率较高。

如果主键无序，比如随机 UUID，可能需要在已有数据中间插入，导致：

- 页分裂
- 数据移动
- 额外磁盘 IO
- 索引维护成本增加

所以 InnoDB 通常推荐使用自增 ID 作为主键。

### 8.5 回表查询

补充：InnoDB 的辅助索引叶子节点通常保存的是主键值，而不是完整行数据。

如果使用辅助索引查到主键后，还需要再根据主键到聚簇索引中查完整数据，这个过程常叫“回表”。

例如：

```sql
select * from student where province = 'guangdong';
```

如果 `province` 是普通二级索引，查询过程可能是：

```text
先在 idx_province 中找到符合条件的主键 id
再根据 id 到主键索引中查完整行
```

## 9. 索引使用实战

### 9.1 创建数据库和数据表

PPT 中的示例：

```sql
-- 创建数据库
create database school;

-- 创建数据表
create table `student` (
    `id` int not null auto_increment,
    `name` varchar(256) not null comment '姓名',
    `sex` varchar(10) not null comment '性别',
    `age` int not null comment '年龄',
    `province` varchar(128) not null comment '省份',
    primary key (`id`)
) comment '学生表';
```

### 9.2 插入数据

```sql
insert into `student`
(`name`, `sex`, `age`, `province`)
values
('wangwu', 'male', 16, 'guangdong'),
('xiaohuang', 'female', 15, 'guangxi');
```

### 9.3 创建普通索引

语法：

```sql
alter table `表名`
add index `索引名` (`字段1`, `字段2`, ..., `字段N`);
```

示例：给 `province` 增加普通索引。

```sql
alter table `student`
add index `idx_province` (`province`);
```

普通索引只提升查询效率，不限制数据重复。

### 9.4 创建唯一索引

语法：

```sql
alter table `表名`
add unique index `索引名` (`字段1`, `字段2`, ..., `字段N`);
```

示例：给 `name` 和 `province` 创建联合唯一索引。

```sql
alter table `student`
add unique index `uniq_idx_name_province` (`name`, `province`);
```

这表示：

```text
同一个 name + province 组合不能重复
```

例如已经有：

```text
name = wangwu
province = guangdong
```

再插入同样组合时会失败：

```sql
insert into `student`
(`name`, `sex`, `age`, `province`)
values
('wangwu', 'male', 19, 'guangdong');
```

报错类似：

```text
ERROR 1062 (23000): Duplicate entry 'wangwu-guangdong'
for key 'student.uniq_idx_name_province'
```

说明唯一索引生效了。

### 9.5 删除索引

语法：

```sql
alter table `表名`
drop index `索引名1`, drop index `索引名N`;
```

示例：

```sql
alter table `student`
drop index `idx_province`,
drop index `uniq_idx_name_province`;
```

删除唯一索引后，之前因为唯一约束失败的数据就可以插入成功：

```text
Query OK, 1 row affected
```

## 11. 查询性能对比实战

### 11.1 没有索引时查询

示例查询：

```sql
select *
from employees
where hire_date = '1989-06-02'
order by birth_date desc
limit 10;
```

PPT 中无索引时查询耗时约：

```text
10 rows in set (0.09 sec)
```

### 11.2 创建索引

给 `hire_date` 创建索引：

```sql
alter table `employees`
add index `idx_hire_date` (`hire_date`);
```

执行结果：

```text
Query OK, 0 rows affected (0.21 sec)
Records: 0  Duplicates: 0  Warnings: 0
```

### 11.3 创建索引后再次查询

再次执行：

```sql
select *
from employees
where hire_date = '1989-06-02'
order by birth_date desc
limit 10;
```

PPT 中建索引后查询耗时约：

```text
10 rows in set (0.01 sec)
```

对比：

| 状态 | 查询耗时 |
| --- | --- |
| 没有索引 | 约 `0.09 sec` |
| 创建 `hire_date` 索引后 | 约 `0.01 sec` |

这个例子说明：合适的索引可以显著减少查询耗时。

## 12. 使用 EXPLAIN 分析查询

`EXPLAIN` 可以用来查看 MySQL 如何执行一条查询语句。

用法：

```sql
explain
select *
from employees
where hire_date = '1989-06-02'
order by birth_date desc
limit 10;
```

### 12.1 建索引后的 EXPLAIN

PPT 中创建 `idx_hire_date` 后，`EXPLAIN` 结果中：

```text
type = ref
possible_keys = idx_hire_date
key = idx_hire_date
rows = 75
Extra = Using filesort
```

重点看：

| 字段 | 含义 |
| --- | --- |
| `possible_keys` | 可能使用的索引 |
| `key` | 实际使用的索引 |
| `rows` | MySQL 预估需要扫描的行数 |
| `type` | 访问类型，能反映查询效率 |
| `Extra` | 额外执行信息 |

这里：

```text
key = idx_hire_date
```

说明本次查询实际使用了 `idx_hire_date` 索引。

```text
rows = 75
```

说明预估只需要扫描约 75 行。

### 12.2 删除索引后的 EXPLAIN

删除索引：

```sql
alter table `employees`
drop index `idx_hire_date`;
```

再次执行 `EXPLAIN`，PPT 中结果显示：

```text
type = ALL
possible_keys = NULL
key = NULL
rows = 299157
Extra = Using where; Using filesort
```

含义：

```text
key = NULL
```

说明没有使用索引。

```text
type = ALL
```

通常表示全表扫描。

```text
rows = 299157
```

说明预估扫描接近整张表的数据量，所以查询较慢。

### 12.3 Extra 中的 Using filesort

补充：`Using filesort` 表示 MySQL 需要额外排序。

在 PPT 的查询中：

```sql
where hire_date = '1989-06-02'
order by birth_date desc
```

只给 `hire_date` 建索引，虽然可以加速筛选条件，但不一定能同时满足 `birth_date` 排序，所以仍然可能出现 `Using filesort`。

如果希望进一步优化排序，可以考虑联合索引，例如：

```sql
alter table employees
add index idx_hire_date_birth_date (hire_date, birth_date);
```

是否合适还要结合实际查询、数据分布和 `EXPLAIN` 判断。

## 13. 最左前缀匹配原则

最左前缀匹配原则是联合索引中非常重要的规则。

PPT 中的定义：

> 使用联合索引查询时，只有查询条件中使用了索引的最左前缀列，才能使用该索引进行查询。

例如创建联合索引：

```sql
create index idx_abc on table_name (a, b, c);
```

它大致相当于创建了：

```text
(a)
(a, b)
(a, b, c)
```

但不等于创建了：

```text
(b)
(c)
(b, c)
```

### 13.1 哪些查询可以使用联合索引？

假设有联合索引 `(a, b, c)`。

| 查询条件 | 是否能较好使用索引 | 说明 |
| --- | --- | --- |
| `where a = 2` | 可以 | 使用最左列 `a` |
| `where a = 2 and b = 3` | 可以 | 使用 `(a, b)` |
| `where a = 2 and b = 3 and c = 4` | 可以 | 使用 `(a, b, c)` |
| `where b = 2` | 不可以直接用完整联合索引 | 缺少最左列 `a` |
| `where b = 2 and c = 3` | 不可以直接用完整联合索引 | 缺少最左列 `a` |

### 13.2 范围查询对联合索引的影响

PPT 中还提到：

```sql
where a = 2 and b = 2 and c > 3
```

可以使用索引。

因为 `a`、`b` 是等值匹配，`c` 是范围匹配，符合从左到右的顺序。

但：

```sql
where a > 1 and b = 1 and c = 2
```

虽然可以使用索引，但通常主要使用的是索引中的 `a`，后面的 `b`、`c` 很难继续用于精确定位。

原因是范围条件会让后续列的有序性不能继续充分用于定位。

可以记：

```text
联合索引从左往右用；
遇到范围查询后，后面的列往往不能继续用于精确定位。
```

### 13.3 为什么要遵守最左前缀？

因为联合索引 `(a, b, c)` 在 B+ 树中是按从左到右的顺序排序的。

比较两个索引值：

```text
(a1, b1, c1)
(a2, b2, c2)
```

排序时：

1. 先比较 `a1` 和 `a2`
2. 如果 `a` 相等，再比较 `b1` 和 `b2`
3. 如果 `b` 相等，再比较 `c1` 和 `c2`

所以，如果查询条件没有 `a`，直接查 `b`，数据库就很难在这棵按 `a` 优先排序的 B+ 树中快速定位。

### 13.4 联合索引列顺序怎么选？

PPT 中提到：创建联合索引时，经常把最常用的列放在最左。

实际设计时通常考虑：

- 查询中最常出现的列
- 等值匹配频率高的列
- 区分度高的列
- 排序、分组中使用的列
- 是否能覆盖多个常见查询

例如常见查询是：

```sql
where user_id = ?
  and status = ?
order by create_time desc
```

可能考虑：

```sql
create index idx_user_status_create_time
on orders (user_id, status, create_time);
```

但最终仍要通过真实数据和 `EXPLAIN` 验证。

## 14. 常见索引设计建议

### 14.1 适合建索引的列

通常适合建索引：

- 经常出现在 `where` 条件中的列
- 经常用于 `join` 的关联列
- 经常用于 `order by` 的列
- 经常用于 `group by` 的列
- 区分度较高的列
- 需要保证唯一性的列

### 14.2 不太适合建索引的列

通常不适合：

- 数据量很小的表
- 频繁更新但很少查询的列
- 区分度很低的列，例如性别
- 大字段，例如很长的 `text`
- 已经被更合适的联合索引覆盖的列

### 14.3 区分度是什么？

区分度可以理解为“这个列能把数据分散开的程度”。

例如：

```text
性别：male/female，区分度低
手机号：几乎每个人不同，区分度高
身份证号：几乎唯一，区分度高
```

区分度越高，索引筛选效果通常越好。

## 15. 常见误区

### 15.1 建了索引就一定会用

不一定。

MySQL 优化器会根据成本选择执行计划。

即使存在索引，也可能因为以下原因不用：

- 查询条件不符合最左前缀
- 函数或表达式破坏索引使用
- 数据量太小，全表扫描更划算
- 条件选择性太差
- 隐式类型转换

### 15.2 索引越多越好

不对。

索引越多：

- 占用磁盘空间越多
- 写入、更新、删除越慢
- 优化器选择成本也可能增加

应该建立必要索引，而不是给每个字段都建索引。

### 15.3 联合索引和多个单列索引完全一样

不一样。

例如：

```sql
create index idx_a on t(a);
create index idx_b on t(b);
```

不等价于：

```sql
create index idx_ab on t(a, b);
```

联合索引可以按组合顺序组织数据，在多条件过滤、排序、覆盖查询等方面更有优势。

### 15.4 `key = NULL` 说明没用索引

在 `EXPLAIN` 中，`key` 表示实际使用的索引。

如果：

```text
key = NULL
```

通常说明本次查询没有使用索引。

## 16. 快速记忆

### 16.1 一句话理解索引

索引是数据库用来快速定位数据的数据结构，类似书的目录。

### 16.2 一句话理解 B+ 树

B+ 树是一种多路平衡搜索树，非叶子节点负责导航，数据集中在叶子节点，叶子节点之间形成有序链表。

### 16.3 一句话理解聚簇索引

聚簇索引把索引和数据存放在一起，InnoDB 的主键索引就是聚簇索引。

### 16.4 一句话理解最左前缀

联合索引必须从最左列开始连续使用，才能充分发挥索引效果。

### 16.5 口诀

```text
索引像目录，查询少扫描。
B+ 树降树高，叶子链表跑范围。
主键是聚簇，二级会回表。
联合看最左，范围后面难精确。
EXPLAIN 看 key，rows 越少越轻巧。
```

## 17. 面试/考试高频点

### 17.1 为什么 MySQL 常用 B+ 树做索引？

因为 B+ 树：

- 树高低，磁盘 IO 少
- 非叶子节点只存关键字，可以放更多索引项
- 叶子节点有序链表，适合范围查询
- 查询性能稳定

### 17.2 B 树和 B+ 树的区别

| 对比项 | B 树 | B+ 树 |
| --- | --- | --- |
| 数据存储 | 每个节点都可以存数据 | 数据只在叶子节点 |
| 非叶子节点 | 存关键字和数据 | 只存关键字和指针 |
| 范围查询 | 通常需要中序遍历 | 叶子节点链表更适合范围查询 |
| 树高 | 相对更高 | 通常更矮 |

### 17.3 主键索引和唯一索引的区别

| 对比项 | 主键索引 | 唯一索引 |
| --- | --- | --- |
| 是否唯一 | 唯一 | 唯一 |
| 是否允许 NULL | 不允许 | 通常允许 |
| 一张表数量 | 只能一个 | 可以多个 |
| InnoDB 中作用 | 聚簇索引 | 辅助索引 |

### 17.4 什么是最左前缀匹配原则？

联合索引 `(a, b, c)` 查询时，必须从最左列 `a` 开始连续匹配，才能充分使用索引。

可以使用：

```text
a
a, b
a, b, c
```

不能直接充分使用：

```text
b
c
b, c
```

### 17.5 如何判断查询是否使用索引？

使用：

```sql
explain select ...
```

重点看：

- `key`：实际使用的索引
- `rows`：预估扫描行数
- `type`：访问类型
- `Extra`：额外信息

如果 `key = NULL`，通常表示没有使用索引。
