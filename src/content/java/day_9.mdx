# Java 集合骨架实现：AbstractCollection / AbstractList / AbstractSet / AbstractQueue / AbstractMap 笔记

---

## 一、什么是“抽象骨架实现”

Java 集合框架里有一组以 `Abstract` 开头的类，例如：

* `AbstractCollection`
* `AbstractList`
* `AbstractSet`
* `AbstractQueue`
* `AbstractMap`

它们的作用可以理解为：

* 已经帮你实现了一部分通用逻辑
* 子类只需要实现少量核心方法
* 就能快速得到一个可用的集合实现

所以它们本质上是：

**集合接口的“骨架实现”**

---

# 二、AbstractCollection

## 1. 作用

`AbstractCollection`：

* 实现了 `Collection` 接口
* 提供了一个 `Collection` 的最小骨架实现

如果你想自己实现一个集合，通常可以继承它，少写很多重复代码。

---

## 2. 最少需要实现什么

如果想做一个**不可变集合**，通常只需要实现：

* `iterator()`
* `size()`

因为很多通用方法它已经帮你写好了，比如：

* `contains`
* `toArray`
* `remove`（依赖迭代器）
* `isEmpty`

---

## 3. contains()

### 核心逻辑

`contains(Object o)` 的实现本质就是：

* 通过迭代器遍历整个集合
* 逐个比较元素是否相等
* 找到就返回 `true`
* 否则返回 `false`

### 关键点

* 如果传入的是 `null`，用 `== null` 判断
* 否则用 `equals()` 判断

所以可以记成：

**AbstractCollection 的很多默认实现，本质就是“基于 iterator() 遍历整个集合”。**

---

## 4. toArray()

### 核心作用

把集合内容转换成数组。

### 核心思路

它的实现重点不是“简单拷贝”，而是要处理这种情况：

* `size()` 预估的大小
* 和真正迭代出来的元素个数
* 可能不一致

所以流程是：

1. 先根据 `size()` 预估创建数组
2. 一边遍历一边填充
3. 如果元素比预期少，就截断
4. 如果元素比预期多，就扩容再继续装

### 你这里最该记住的点

不是去背扩容细节，而是记住：

**toArray() 会以 `size()` 为初始预估，但真正结果以迭代器遍历到的元素为准。**

---

## 5. add()

`AbstractCollection` 里的 `add(E e)` 默认**不是抽象方法**，而是直接抛异常：

```java
UnsupportedOperationException
```

### 为什么这样设计

因为：

* 有的集合可变
* 有的集合不可变

所以默认不给出“可修改”的实现，而是让子类自己决定是否支持添加。

---

## 6. remove()

`remove(Object o)` 的默认实现本质也是：

* 通过 `iterator()` 遍历
* 找到目标元素后
* 调用迭代器的 `remove()`

### 关键点

它删除的是：

**第一个匹配到的元素**

### 重要前提

这个实现能否正常工作，依赖：

* 迭代器是否正确实现了 `remove()`

如果迭代器不支持删除，那么这里也会失败。

---

# 三、AbstractList

## 1. 作用

`AbstractList`：

* 继承 `AbstractCollection`
* 实现 `List` 接口
* 是 `List` 的骨架实现

---

## 2. 它是不是链表

不是。

`AbstractList` 不是链表，也不是数组，它只是一个：

**面向 List 的抽象骨架类**

---

## 3. 为什么还有 AbstractSequentialList

因为 `List` 有两类典型底层结构：

* 数组型（如 `ArrayList`）
* 链表型（如 `LinkedList`）

如果链表也直接按 `AbstractList` 那种“偏索引访问”的思路来实现，性能会不好。
所以 Java 又提供了：

* `AbstractSequentialList`

专门给顺序访问结构使用。

---

## 4. 实现一个 AbstractList 最少要做什么

### 不可变 List

通常只需要实现：

* `get(int index)`
* `size()`

这样就已经能得到一个只读 List。

---

### 可变 List

如果还想支持修改，一般要继续重写：

* `set(int index, E element)`
* `add(int index, E element)`
* `remove(int index)`

否则默认很多修改操作都会抛：

```java
UnsupportedOperationException
```

---

## 5. indexOf()

### 作用

查找元素第一次出现的位置。

### 核心思路

它内部本质是：

* 用 `ListIterator` 从前往后遍历
* 找到第一个相等元素就返回索引
* 找不到返回 `-1`

---

## 6. lastIndexOf()

### 作用

查找元素最后一次出现的位置。

### 核心思路

它内部本质是：

* 从尾部开始向前遍历
* 找到第一个匹配的就返回索引
* 找不到返回 `-1`

---

## 7. clear()

`clear()` 默认通过：

```java
removeRange(0, size())
```

来完成。

### removeRange()

默认实现本质上是：

* 先拿到区间起点的迭代器
* 再逐个删除指定范围元素

### 注意

很多高性能子类会重写 `removeRange()`，因为默认逐个删不一定高效。

---

## 8. addAll(int index, Collection c)

### 核心逻辑

本质就是：

* 先做范围检查
* 再把传入集合元素一个个插入进去

所以默认实现偏通用，**不一定高效**。

---

## 9. 内部迭代器

`AbstractList` 内部有两个重要迭代器实现：

### Itr

实现的是 `Iterator`

内部常见字段有：

* `cursor`
* `lastRet`
* `expectedModCount`

---

### ListItr

继承自 `Itr`，实现 `ListIterator`

比普通 `Iterator` 多了：

* 向前遍历
* 定位索引
* 双向移动

---

## 10. expectedModCount 是什么

这是用来做：

**快速失败（Fail-Fast）检查**

如果遍历过程中，集合结构被外部非法修改，就可能抛出：

```java
ConcurrentModificationException
```

这个字段你记住作用就行，不用死背细节。

---

## 11. subList()

`subList(fromIndex, toIndex)` 返回的不是独立副本，而是：

**原 List 的一个视图（view）**

### 这点非常重要

对 `subList` 的修改，通常会影响原列表；
对原列表的结构性修改，也可能影响 `subList`。

所以可以记成：

**subList 返回的是“窗口”，不是“拷贝”。**

---

## 12. RandomAccess

`subList()` 里会判断是否实现了 `RandomAccess`。

### RandomAccess 是什么

它是一个**标记接口**，表示这个 List 适合随机访问，也就是：

* `get(index)` 很快

典型例子：

* `ArrayList` 实现了 `RandomAccess`
* `LinkedList` 没有实现

### 经验结论

对实现 `RandomAccess` 的 List：

* 普通 `for` 循环通常更合适

对链表型 List：

* 迭代器通常更合适

---

## 13. equals()

`AbstractList` 的 `equals()` 比较的是：

* 两个 List 是否元素个数相同
* 且对应位置元素都相等

所以 `List` 的相等性是：

**顺序相关的**

例如：

```java
[1, 2, 3]
```

和

```java
[3, 2, 1]
```

即使元素一样，也不相等。

---

## 14. hashCode()

`AbstractList` 的 `hashCode()` 是按顺序一个个元素计算出来的。

### 你这里有个容易混的点，纠正一下

`hashCode()` **不是给 `equals()` 用的**。
更准确地说：

* `equals()` 是判断逻辑相等
* `hashCode()` 是为了配合哈希容器使用

例如：

* `HashSet`
* `HashMap`

### 正确认识

* `hashCode` 不相等，两个对象通常不相等
* `hashCode` 相等，两个对象**不一定**相等，还要再看 `equals`

所以 `hashCode()` 的主要用途不是“替代 equals”，而是：

**先快速缩小范围，再配合 equals 精确判断**

---

# 四、AbstractSet

## 1. 作用

`AbstractSet`：

* 继承 `AbstractCollection`
* 实现 `Set` 接口
* 是 `Set` 的骨架实现

---

## 2. 特点

它主要重写了：

* `equals()`
* `hashCode()`
* `removeAll()`

---

## 3. Set 的 equals 语义

和 `List` 不同，`Set` 的相等性不看顺序，只看：

* 元素是否相同

例如：

```java
[1, 2, 3]
```

和

```java
[3, 2, 1]
```

如果是 `Set`，它们可以相等。

---

# 五、AbstractQueue

## 1. 作用

`AbstractQueue`：

* 继承 `AbstractCollection`
* 实现 `Queue` 接口
* 是队列的骨架实现

---

## 2. 作用理解

它主要帮你把一些通用的 Queue 行为先实现好，子类只需要关注核心队列操作。

比如：

* `offer`
* `poll`
* `peek`

这些核心方法由子类决定具体实现。

---

# 六、AbstractMap

## 1. 作用

`AbstractMap`：

* 实现 `Map` 接口
* 是 `Map` 的骨架实现

---

## 2. 核心思想

`AbstractMap` 很多逻辑都建立在：

```java
entrySet()
```

之上。

也就是说：

**只要子类能提供 entrySet()，很多 Map 通用行为就能工作。**

---

## 3. 常见依赖 entrySet() 的方法

例如：

* `size()`
* `containsKey()`
* `containsValue()`
* `get()`
* `remove()`
* `clear()`
* `keySet()`
* `values()`

很多默认实现，本质上都是遍历 `entrySet()`。

---

## 4. put()

`put(K key, V value)` 在 `AbstractMap` 中默认没有真正实现，通常直接抛异常。

因为：

* 有的 Map 可修改
* 有的 Map 不可修改

所以让子类自己决定。

---

## 5. remove()

默认思路也是：

* 先遍历 `entrySet()`
* 找到匹配 key 的 entry
* 再删除

---

## 6. clear()

默认实现本质就是：

```java
entrySet().clear()
```

---

## 7. keySet() 和 values()

`AbstractMap` 会基于 `entrySet()` 封装出两个视图：

* `keySet()`：键的集合视图
* `values()`：值的集合视图

注意，它们通常也是：

**视图，不是独立副本**

---

## 8. SimpleEntry

`AbstractMap` 里有一个常见内部类：

* `SimpleEntry`

它表示一个**可变键值对**

特点：

* key / value 组成一个 Entry
* value 可以通过 `setValue()` 修改

---

# 七、这一部分最该记住的点

## AbstractCollection

* `Collection` 的骨架实现
* 最少常实现：`iterator()` 和 `size()`
* 很多默认方法本质依赖迭代器遍历

## AbstractList

* `List` 的骨架实现
* 不是链表
* 不可变 List 通常只要 `get()` 和 `size()`
* `subList()` 返回的是视图，不是副本
* `List` 的 `equals()` 看顺序

## AbstractSet

* `Set` 的骨架实现
* `Set` 的相等性不看顺序

## AbstractQueue

* `Queue` 的骨架实现
* 子类重点实现核心队列操作

## AbstractMap

* `Map` 的骨架实现
* 很多默认实现都建立在 `entrySet()` 上
* `keySet()`、`values()` 通常是视图

## hashCode

* 不是“给 equals 用的”
* 是为了哈希容器更高效地定位和比较
* `hashCode` 相等，不代表对象一定相等
* 真正判等仍然要看 `equals`
