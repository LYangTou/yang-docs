# 📚 DRF 通用视图 (Generic Views) 全能实战笔记



**核心概念**：通用视图是对 API 开发中常见的“增删改查 (CRUD)”模式的高度抽象，旨在消除重复的样板代码。DRF 的通用视图体系主要由**三个层级**像搭积木一样组合而成：
1. **`GenericAPIView`**：提供核心数据处理能力（查询集、序列化器、基础过滤、分页）。
2. **`Mixins`（混入类）**：提供具体的 CRUD 动作逻辑（如列表、创建、删除），但不绑定 HTTP 请求。
3. **`Concrete Views` (具体视图类)**：开箱即用的视图，将上述两者结合，并直接绑定到 GET、POST 等具体的 HTTP 方法。

### 0. 🛠️ 前置准备：贯穿全篇的业务场景代码
为了让后续代码易于理解，我们假设正在开发一个**博客文章（Article）API**。以下是依赖的模型和序列化器：

```python
# models.py
from django.db import models
from django.contrib.auth.models import User

class Article(models.Model):
    title = models.CharField(max_length=100)
    content = models.TextField()
    author = models.ForeignKey(User, on_delete=models.CASCADE)
    is_published = models.BooleanField(default=False)
    slug = models.SlugField(unique=True) # 英文简写，用于自定义 URL

# serializers.py
from rest_framework import serializers

class ArticleSerializer(serializers.ModelSerializer):
    class Meta:
        model = Article
        fields = '__all__'
```

## ⚙️ 一、基石层：GenericAPIView 核心配置与方法

`GenericAPIView` 是所有通用视图的父类，负责定义“去哪查数据”和“如何转换数据”。

### 1. 核心属性配置
* **`queryset`**: 视图操作的基础模型实例集合（数据源）。
* **`serializer_class`**: 用于验证输入、反序列化和序列化输出的类。
* **`lookup_field`**: 详情页中用于查找单个对象的模型字段，默认是 `'pk'`（主键）。
* **`Workspace_kwarg`**: URL 路由中用于匹配对象的参数名。若不设置，默认与 `lookup_field` 同名。

**💡 实战代码：自定义详情页查找字段**
场景：详情页 URL 不使用数字 ID，而是使用 `slug`（如 `/api/articles/my-first-post/`）。
```python
from rest_framework import generics

class ArticleSlugDetailView(generics.RetrieveAPIView):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer
    
    # 告诉 DRF 去数据库里匹配 'slug' 字段，而不是默认的 'id'
    lookup_field = 'slug'          
    
    # 告诉 DRF 从 URL 路由里提取名为 'post_slug' 的变量值
    lookup_url_kwarg = 'post_slug' 
```

### 2. 核心方法重写 (动态逻辑与性能调优)
静态属性往往不够用，通过重写方法可以实现动态业务逻辑。
* **`get_queryset(self)`**: 🌟 高频使用。用于实现基于用户/参数的动态数据过滤，以及利用 `select_related`/`prefetch_related` 解决数据库 N+1 查询性能问题。

**💡 实战代码：动态过滤与 N+1 性能优化**
场景：接口只返回已发布的文章；若 URL 带 `?author_id=x` 则过滤该作者；且需一次性查出关联的作者信息。
```python
class ArticleListView(generics.ListAPIView):
    serializer_class = ArticleSerializer

    def get_queryset(self):
        # 1. 基础查询集：只查已发布，并用 select_related 提前 JOIN 查出作者信息(解决N+1)
        queryset = Article.objects.select_related('author').filter(is_published=True)
        
        # 2. 动态过滤：获取 URL 参数
        author_id = self.request.query_params.get('author_id')
        if author_id is not None:
            queryset = queryset.filter(author_id=author_id) # 追加过滤条件
            
        return queryset
```

## 🧩 二、动作层：Mixins 混入类与生命周期钩子

Mixins 位于 `rest_framework.mixins`，它们不处理 HTTP 请求，只提供纯粹的动作方法：
* `ListModelMixin` (.list) / `CreateModelMixin` (.create)
* `RetrieveModelMixin` (.retrieve)
* `UpdateModelMixin` (.update / .partial_update)
* `DestroyModelMixin` (.destroy)

### 🪝 关键生命周期钩子 (Hooks)
在执行增、改、删时，Mixins 预留了钩子方法，允许我们在数据落库前进行拦截和修改：
* `perform_create(self, serializer)`
* `perform_update(self, serializer)`
* `perform_destroy(self, instance)`

**💡 实战代码：拦截保存动作注入隐式数据**
场景：发布文章时，防伪造客户端传入的作者 ID，由后端自动绑定当前登录用户为文章作者。
```python
class ArticleCreateView(generics.CreateAPIView):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer

    def perform_create(self, serializer):
        # 强行塞入当前发起请求的用户对象 (要求视图已配置正确的身份认证)
        serializer.save(author=self.request.user)
```

## 🚀 三、应用层：具体视图类 (Concrete Views)

位于 `rest_framework.generics`，是日常开发最常继承的类。它们把前两层组合起来，并暴露给具体的 HTTP 方法。如果你的需求是标准 CRUD，代码可以极简。

**常用列表/创建视图：**
* `ListAPIView` (GET)
* `CreateAPIView` (POST)
* `ListCreateAPIView` (GET, POST)

**常用详情视图：**
* `RetrieveAPIView` (GET)
* `UpdateAPIView` (PUT, PATCH)
* `DestroyAPIView` (DELETE)
* `RetrieveUpdateDestroyAPIView` (GET, PUT, PATCH, DELETE) -> **全能详情视图**

**💡 实战代码：极简的标准 CRUD 接口**
```python
# 1. 文章列表与发布接口 (处理 GET 和 POST)
class ArticleListCreateView(generics.ListCreateAPIView):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer

# 2. 文章详情、修改与删除接口 (处理 GET, PUT, PATCH, DELETE)
class ArticleDetailView(generics.RetrieveUpdateDestroyAPIView):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer
```

## 🛠️ 四、高阶定制与扩展架构

当框架自带的组合不够用时，DRF 鼓励你创造自己的“积木”。

### 1. 打造专属的自定义 Mixin
**💡 实战代码：实现多字段联合查找**
场景：有时需要同时通过 `author_id` 和 `slug` 两个条件来精确定位一篇文章。
```python
from django.shortcuts import get_object_or_404

# 步骤A：定义你自己的 Mixin
class MultipleFieldLookupMixin:
    """允许通过定义 lookup_fields 列表来实现多字段匹配对象的 Mixin"""
    def get_object(self):
        queryset = self.filter_queryset(self.get_queryset())
        
        filter_kwargs = {}
        # 遍历自定义的多字段列表，从 URL kwargs 中提取对应的值
        for field in self.lookup_fields:
            if self.kwargs.get(field):
                filter_kwargs[field] = self.kwargs[field]
                
        # 解包字典，传入多个查询条件
        obj = get_object_or_404(queryset, **filter_kwargs)
        self.check_object_permissions(self.request, obj)
        return obj

# 步骤B：将你的 Mixin 与官方的基础视图组合使用
class CustomArticleDetailView(MultipleFieldLookupMixin, generics.RetrieveAPIView):
    queryset = Article.objects.all()
    serializer_class = ArticleSerializer
    lookup_fields = ['author_id', 'slug'] # 这是给自定义 Mixin 读取的属性
```

### 2. 其他重要知识点
* **构建专属 Base Class**: 如果上面的 `MultipleFieldLookupMixin` 在项目中被高频使用，你可以定义一个 `class BaseMultipleLookupView(MultipleFieldLookupMixin, generics.RetrieveAPIView): pass`，让全团队直接继承它。
* **PUT 请求的规范 (PUT as Create)**: DRF 3.0 开始，对不存在的资源发起 PUT 请求默认会返回 `404 Not Found`（而不是像老版本那样尝试创建新资源），这让接口语义更明确、更安全。
* **第三方扩展**: 若需在一个接口中同时返回多个不同模型的数据（如做 App 首页数据聚合），推荐使用官方提及的第三方包 `Django Rest Multiple Models`。
